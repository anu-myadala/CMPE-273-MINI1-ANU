A Counterintuitive Discovery in Optimization

We Added 8 CPU Cores. It Barely Helped.

Analyzing 18 Million NYC 311 Records (12 GB) via C++ & OpenMP
❌ The Expectation
"Just throw more threads at it!"
⚙️ 8 OpenMP Threads
Only 1.2× Faster
Query time dropped from 1,159 ms (serial) to 933 ms.

Conclusion: 8x the CPU power yielded negligible results. The CPU was not the bottleneck.
✅ The Real Solution
"Change how the data is stored in memory."
🚀 Structure of Arrays (SoA)
119× Faster
Query time plummeted from 1,159 ms to 9.7 ms.

Conclusion: Vectorizing the memory layout solved the true bottleneck: Cache Starvation.

The CPU wasn't slow. It was starving.

In an Array-of-Structures (AoS), a 500-byte record contains a single 4-byte date. To scan 18M dates, we loaded 9 GB of useless fields into the L3 cache.

Structure-of-Arrays (SoA) packs all target fields together. We turned a 9 GB scattered read into a 72 MB contiguous read that fits entirely in the cache.
