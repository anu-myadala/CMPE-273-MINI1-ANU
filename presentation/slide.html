Data Layout Beats Parallelism:
SoA Achieves 125× Speedup — OpenMP Alone Only 20%

CMPE-273 Mini 1  |  NYC 311 Service Requests 2020–2025 (18M records, 12 GB)  |  Anu Myadala

The Finding

125×
faster date-range queries
in Phase 3 (SoA) vs Phase 1 (Serial AoS)

A date-range query over 18 million records only needs created_ymd — 4 bytes per record. In Array-of-Structures (AoS), each record is ~500 bytes wide: the CPU must load and discard 98% of every cache line to reach those 4 bytes.

Structure-of-Arrays (SoA) stores all dates in a contiguous uint32_t[] array (72 MB) that fits entirely in L3 cache. The query drops from 1,196 ms to 9.6 ms.

Why this matters beyond this assignment: Urban analytics systems — real-time 311 dashboards, GIS services, emergency dispatch — run filter queries continuously. A 125× latency reduction means sub-10ms responses on 18M live records, enabling truly interactive city data exploration.

Key insight: OpenMP on AoS (Phase 2) yields only ~20% improvement. SoA (Phase 3) achieves 29–125× speedup because it eliminates 98% cache-line waste. Memory layout matters more than threading.
